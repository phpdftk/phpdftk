title: Cross-browser WPT sweep

The cross-browser PDF oracle renders WPT fixtures through three real browser engines — Blink (Chromium), Gecko (Firefox), and WebKit — and asserts our renderer's output matches the engines' consensus. The sweep is how we measure that against the entire WPT corpus.

This page covers the **container-hosted sweep** that uses persistent engine daemons. The legacy per-fixture path (`scripts/cross-browser/render.mjs`) still works and is the right choice for one-off renders during dev, but for a full-corpus sweep the persistent-daemon path is 5–10× faster.

Background: the oracle's *what* lives in [docs/plans/cross-browser-oracle.md](https://github.com/phpdftk/phpdftk/blob/main/docs/plans/cross-browser-oracle.md); the container architecture is in [docs/plans/cross-browser-container-sweep.md](https://github.com/phpdftk/phpdftk/blob/main/docs/plans/cross-browser-container-sweep.md).

## Why daemons

A one-shot `node render.mjs chromium fixture.html` cold-launches Chromium (~400 ms) before doing the actual render (~50–500 ms). Over the ~28 000 fixtures in a `--scope=rendering --skip-testharness` sweep × 3 engines, that's about 9 hours of pure process startup wasted.

The daemon launches the browser once at the start of the sweep and recycles `BrowserContext`s from a pool. Startup cost amortizes to nothing; the only per-fixture cost is the actual render.

## Bring it up

```sh
docker compose up -d --wait chromium firefox webkit
curl http://127.0.0.1:9101/status   # chromium
curl http://127.0.0.1:9102/status   # firefox
curl http://127.0.0.1:9103/status   # webkit
```

The first build is ~3 min per engine. Subsequent runs hit the docker layer cache and come up in seconds. The compose file lives at the repo root.

Bring-up times after the container starts:

| Engine | Cold ready | Why |
|---|---|---|
| chromium | ~1 s | Playwright launches Chromium eagerly |
| firefox | ~3–5 s | Geckodriver session + Firefox profile creation |
| webkit | ~2–3 s | Xvfb + GLib main loop + WebKitWebContext bring-up |

The `start_period` on each service's healthcheck accounts for the difference.

To stop everything:

```sh
docker compose down
```

## Run the sweep against the daemon

Tell `wpt cb-sweep` (or `wpt cross-browser`) to route through the daemon cluster:

```sh
composer wpt -- cb-sweep \
    --scope=rendering \
    --skip-testharness \
    --engines=chromium,firefox,webkit \
    --daemon-base=http://127.0.0.1:910
```

`--daemon-base` is a single URL prefix. The oracle appends an engine offset:

| Engine | Port |
|---|---|
| chromium | 9101 |
| firefox | 9102 |
| webkit | 9103 |

So `--daemon-base=http://127.0.0.1:910` resolves to `9101`, `9102`, `9103` for the three engines respectively. The corresponding compose services have to be up; engines whose daemon doesn't answer are skipped silently.

When `--daemon-base` is unset, the oracle falls back to the legacy per-fixture path. Dev hosts that don't want to run Docker keep working without changing anything.

## Cache

Both the host PHP shards and the daemon share `var/wpt/browser-cache/` via a single bind mount. The cache is keyed on `sha256(fixture_bytes) + cache_generation + engine` — same scheme either path uses. Writes are atomic (`tmp + rename`). A cache hit short-circuits the daemon roundtrip entirely; on warm cache, a full sweep finishes in seconds.

Bump the cache generation when any browser version moves — there's a single `cacheGeneration` knob in `BrowserOracle::__construct()`.

## Protocol surface

Each engine daemon speaks the same HTTP/JSON protocol on its loopback port:

```
GET  /status   → 200 { engine, version, ready, pool: { size, in_flight, queued } }
POST /render   → 200 { pdf_bytes_base64, ms, from_cache }
                 4xx { error }  on bad fixture / missing field
                 408 { error }  on render timeout
                 502 { error }  during a browser-crash window
POST /flush    → 204             drain in-flight, restart browser
```

`POST /render` request body:

```json
{
  "fixture":    "/wpt/css/css-flexbox/align-items-001.html",
  "cache_key":  "<first 64 hex chars of sha256 hash>",
  "viewport":   { "width": 816, "height": 1056 },
  "timeout_ms": 60000
}
```

The `fixture` path is interpreted **inside the container**, where the WPT corpus is bind-mounted at `/wpt/`. The daemon rejects any path that doesn't resolve into the WPT mount.

## Daemon lifecycle

All three daemons eagerly bring up their browser at startup so `/status` flips to `ready: true` before any request lands — the PHP `BrowserOracle::isAvailable()` probe would otherwise reject a perfectly usable daemon on the first call.

- **Chromium** launches via Playwright. If the browser process dies — OOM, segfault, anything — the daemon catches Playwright's `disconnected` event and the next request triggers a relaunch.
- **Firefox** is driven over geckodriver's WebDriver protocol. The daemon spawns geckodriver once, creates a single Firefox session, and pre-allocates a window pool via `POST /session/:id/window/new` (`type: "tab"`). Each `/render` claims a window handle from the free-list, navigates via `POST /session/:id/url`, prints via `POST /session/:id/print`, and returns the handle. If geckodriver exits, the next request lazily re-ensures the whole stack.
- **WebKit** runs entirely in-process via webkit2gtk + PyGObject. The daemon spawns under `xvfb-run` to give WebKit a display target, then drives a single `WebKitWebView` via the GLib main loop. Each `/render` posts work to the main thread via `GLib.idle_add`, the load-finished and print-finished signals drive completion, and a threading.Event signals back to the HTTP handler thread. There's no browser child process to restart — webkit2gtk is a library inside the daemon process — so a crash takes the whole daemon down and the container restart-policy revives it.

In all three cases, in-flight requests during a crash window receive a 502 and the orchestrator can retry. `POST /flush` does a controlled restart (or for webkit, bumps the generation counter for protocol parity).

### Engine-specific quirks worth knowing

- **Geckodriver serialises session commands**: multiple concurrent `/render` calls on the same Firefox session don't actually run in parallel. The window-pool design is forward-looking — it pipelines the navigation step (window B can load page B's URL while window A is being printed), which is a real win even though print itself stays sequential.
- **WebKit currently uses a single WebView (pool size = 1)**: webkit2gtk widgets have strict main-thread affinity and need to be created and driven from the GLib main loop. Multi-view pooling is a real follow-up but non-trivial — pull requests welcome. The persistent-process benefit (no cold launch per fixture) is still the dominant win even with serialised renders.
- **WebKit needs `GTK_PRINT_BACKENDS=file` and an explicit `Print to File` printer name**: GTK's default print backend tries to enumerate CUPS printers and fails with "Printer not found" in containers without cupsd. The Dockerfile sets the env var; the daemon pins the printer name in `Gtk.PrintSettings.set_printer()`. Both are required.
- **WebKit PyGObject quirk**: `WebKit2.PrintOperation.print` is exposed as `print_` because the unsuffixed name collides with Python's `print` builtin. The daemon calls `print_op.print_()`.

## Troubleshooting

- **`docker compose up -d --wait <service>` hangs** — the healthcheck waits 30 s (chromium), 60 s (firefox), or 45 s (webkit) on `start_period` before counting failures. If the build itself is slow, run `docker compose up <service>` (no `-d`) once to see the build logs.
- **chromium `/status` returns ready: false right after start** — fixed in the daemon (Chromium launches eagerly), but if it ever regresses, fire one `POST /render` to warm it.
- **firefox `/status` returns ready: false right after start** — geckodriver/session creation failed. Tail `docker logs phpdftk-firefox-daemon` for the geckodriver stderr; the most common cause is a Firefox binary mismatch with the geckodriver version.
- **webkit `/status` returns 503 with "missing state"** — usually means xvfb didn't start. Tail `docker logs phpdftk-webkit-daemon` for the `xvfb-run` stderr; the most common cause is missing `xauth` (already installed in the Dockerfile, but worth checking if you've forked the image).
- **webkit `/render` returns 500 with "print failed: Printer not found"** — the GTK file print backend didn't initialise. Confirm `GTK_PRINT_BACKENDS=file` is set in the container env AND that the daemon calls `settings.set_printer("Print to File")` — both are required.
- **`/render` returns 400 "fixture must be inside /wpt"** — the path you sent isn't under the WPT bind mount. Double-check the mount in `compose.yaml` and that you're using the in-container `/wpt/...` path, not the host path.
- **Daemon eats memory** — likely a leak in the browser-side process pool. Container memory cap is 4 GB per service; the OOMKiller bounces the daemon and your sweep continues. Track at issue [#29](https://github.com/phpdftk/phpdftk/issues/29).
