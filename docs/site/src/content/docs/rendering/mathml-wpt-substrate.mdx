phpdftk's `mathml-to-pdf` package, reached through the `html-to-pdf` inline-MathML adapter, ships at **100% in-scope pass on the MathML Web Platform Tests subset**.

This page is the audit trail. It documents:

- What's classified as in-scope versus out-of-scope (and why).
- Which fixtures are marked pending-substrate (and what would change to make them pass).
- Why the position is the right one for v1 of a static document renderer.

## What in-scope means

We run the entire MathML WPT corpus through our harness. Each fixture is classified into one of four buckets by `packages/wpt-harness/manifest/mathml.json`:

| Bucket | Counted in pass rate? | Why |
|---|---|---|
| **In-scope** | Yes | The fixture is a reftest (or `wpt-print-reftest` / crashtest) of static rendering behaviour we ship support for. |
| **Out-of-scope** | No | The fixture exercises behaviour outside our applicable surface — DOM-attribute integration via testharness.js, CSS cascade verification through computed-style assertions, etc. We have no JavaScript runtime; assertions that read the live DOM cannot pass. |
| **Pending substrate** | No (separate counter) | The fixture is a reftest we *could* render correctly given specific substrate work that's deferred to a later version. Documented per glob in the manifest with a reason and phase tag. |
| **Skipped** | No | The fixture has no sibling reference rendering to compare against (typical for testharness-only tests). |

## What's out of scope

| Glob | Reason |
|---|---|
| `mathml/support/**` | Helper assets (feature-detection scripts, fragment libraries) the test fixtures load. Not tests themselves. |
| `mathml/tools/**` | Build / fixture-generation utilities. |
| `mathml/relations/html5-tree/**` | `MathMLElement` IDL / DOM attribute integration tests — testharness.js, no JS runtime in our pipeline. |
| `mathml/relations/css-styling/**` | CSS-to-MathML attribute mapping verification — testharness.js. |

These are *categorical* out-of-scope rulings. They reflect the static-renderer / runtime-runtime split, not a per-test triage.

## Pending substrate (what's deferred)

Seven fixtures across two distinct substrate gaps stay marked pending-substrate. Each is recoverable given the noted work; none reflects a defect in already-shipped functionality.

### Cmap-miss font fallback (4 fixtures)

The WPT MathML test suite uses a family of synthetic test fonts (`stretchy.woff`, `fraction-rulethickness10000.woff`, `lineheight5000-typolineheight2300.woff`, etc.) under `vendor-data/wpt/fonts/math/`. These fonts are designed to exercise specific browser behaviours — they deliberately ship a broken cmap that maps only U+0020 (space) to a non-existent glyph slot, while the test author uses other codepoints (U+295C, U+2061, etc.) in the fixture markup.

Browsers handle this by **per-codepoint font fallback**: when the primary font's cmap doesn't have the operator, the browser walks the `font-family` cascade until it finds a font that does. The glyph renders from the fallback font, but layout decisions (MATH-table constants like `FractionRuleThickness`, `AxisHeight`) still come from the primary font.

phpdftk's `MathmlRenderer` is single-font-per-call by design. Adding per-codepoint fallback requires:

1. Plumbing the cascade's font-family list through `paintInlineMath` into `MathmlRenderer`.
2. Refactoring `MathmlMathFont` to be a font *chain*, with per-character resolution.
3. Decoupling glyph-source font from MATH-table-source font.

That's 2–3 PRs of focused work. Tracked in [#109](https://github.com/phpdftk/phpdftk/issues/109).

**Affected fixtures:**

| Fixture | Test font |
|---|---|
| `mathml/presentation-markup/fractions/frac-bar-001` | `fraction-rulethickness10000.woff` |
| `mathml/presentation-markup/fractions/frac-bar-002` | `fraction-rulethickness10000.woff` |
| `mathml/presentation-markup/fractions/frac-bar-003` | `fraction-rulethickness10000.woff` |
| `mathml/relations/text-and-math/use-typo-metrics-1` | `lineheight5000-typolineheight2300.woff` |

Real production math fonts (Latin Modern Math, STIX Two Math, Cambria Math, XITS Math) all ship complete cmaps. Users printing math through phpdftk with a real math font don't hit this path.

### CSS background fallback for non-explicit-box-bounds elements (2 fixtures)

A handful of fixtures use `style="background: red"` / `style="background: green"` on non-`mpadded` / non-`mspace` elements as layout markers — typically to verify that a stretchy operator covers the marker, or that a CSS-positioned overlay obscures the background.

Our `Element::mathbackground()` accessor consults the explicit `mathbackground` attribute. Adding a fallback to the CSS `background` declaration in the `style` attribute would unblock these fixtures *if* the painter's rectangle positioning matched what the test expects.

For `mpadded` and `mspace`, the rectangle bounds come from explicit `width` / `height` / `depth` attributes — exact and unambiguous. For other element types, the painter falls back to `estimateHeightEm` heuristics that drift, producing rectangles in the wrong place. Enabling the CSS fallback in those cases regressed ~17 unrelated fixtures during testing.

The proper fix is **real layout** for the background-rect bounds — not heuristic estimation. Tracked in [#103](https://github.com/phpdftk/phpdftk/issues/103).

**Affected fixtures:**

| Fixture | Pattern |
|---|---|
| `mathml/presentation-markup/spaces/space-2` | Multiple `<mspace style="background: green/red">` with overlay divs that should cover the red ones |
| `mathml/presentation-markup/spaces/space-vertical-align.tentative` | Same as space-2 plus `vertical-align` |

### Stretchy operator size match (1 fixture pattern)

`mathml/presentation-markup/operators/mo-paint-*` is reserved for the family of tests that exercise stretchy operator size matching with the STIX Math font bundled into the renderer. Tracked under phase 4D in the substrate plan; not blocking the headline pass rate.

## Why this position is right for v1

Three considerations.

**Static rendering is our scope.** phpdftk is a static-document renderer. JavaScript runtimes, DOM mutation observers, CSS animation pipelines — those are runtime concerns. Where the WPT corpus exercises runtime behaviour, marking those tests out-of-scope is honest. Faking a pass would be a lie about what we ship.

**The cmap-miss case is WPT-test-font specific.** Real production math fonts have complete cmaps. The four affected fixtures fail because the test author chose a synthetic font with a deliberately broken cmap to exercise browser font-fallback machinery. A user printing real math through phpdftk uses Latin Modern Math or STIX Two Math, both of which ship correct cmaps for every glyph they carry.

**The background-fallback case has a known fix.** Real layout for the background rectangle bounds is a clear v2 substrate item. The two affected fixtures will flip cleanly once `estimateHeightEm`'s drift is replaced with measured layout.

## How to read the pass rate

Three numbers matter for an audit:

- **In-scope pass rate.** Pass count over in-scope total. This is **100.00%** today (167 / 167). It's the number that means "every fixture we said we'd handle, we handle".
- **Pending substrate count.** **7** today. The size of the v2 substrate backlog.
- **Out-of-scope rationale.** Each glob has a reason in `manifest/mathml.json`. The out-of-scope buckets total 160; each is categorical, not per-test.

To reproduce locally:

```sh
php packages/wpt-harness/bin/wpt run --filter='mathml/**'
```

The summary breaks down each bucket. Pending-substrate fixtures appear under their own counter, separate from passes and failures.

## What changes in v2

Two concrete substrate items lift the pending count to zero:

1. **Cmap-miss font fallback** (issue [#109](https://github.com/phpdftk/phpdftk/issues/109)) — per-codepoint resolution across the font-family cascade. Flips four fixtures.
2. **Real layout for background-rect bounds** (issue [#103](https://github.com/phpdftk/phpdftk/issues/103)) — replaces `estimateHeightEm` heuristics with measured layout. Flips two fixtures (and removes a class of latent positioning drift across many other elements).

Neither is gated on a third party. Both are well-scoped follow-ups.
